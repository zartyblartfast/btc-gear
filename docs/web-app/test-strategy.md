# btc-gear Web App Test Strategy

Status: Draft for AI-assisted implementation
Date: 2026-05-29
Audience: AI coding agents and human reviewers

## 1. Testing principle

Tests must be honest. They must prove financial and strategy behavior, not merely make the suite green.

A test is honest if it would fail when a user-facing strategy promise is violated.

Examples of dishonest tests:
- Testing that a function returns whatever fixture was generated by the same function.
- Only testing flat BTC paths.
- Only testing happy paths where all strategies look safe.
- Snapshot-testing a chart while ignoring wrong underlying data.
- Testing implementation details instead of outcomes.

## 2. Required test layers

### Layer 1: arithmetic unit tests

Test pure formulas:
- collateral value
- LTV
- liquidation price
- drop-to-liquidation
- max safe debt
- available safe draw
- interest accrual
- net equity
- net BTC after debt
- ARVA annuity payment

### Layer 2: annual lifecycle tests

Must prove:
- interest accrues only on starting debt
- new borrowing in year N accrues interest in year N+1
- ending debt carries to next year
- safety capacity is calculated after interest and before new draw
- liquidation status is detected

### Layer 3: strategy concept tests

These are mandatory and more important than low-level arithmetic tests.

Fixed Draw:
- funds full draw when safe capacity permits
- caps draw when safety capacity is lower
- records skipped income

Supplemental Guardrail:
- draws full desired income in green years
- reduces income when full draw would breach buffer
- draws zero when no safe capacity exists
- resumes income when buffer recovers
- never intentionally breaches required buffer

ARVA:
- recalculates draw from current net equity and remaining horizon
- income rises when net equity improves, all else equal
- income falls when net equity falls, all else equal
- terminal reserve reduces spendable equity
- safety cap overrides raw ARVA draw

ARVA Guardrails:
- caps annual increases
- caps annual decreases in normal safe conditions
- safety override beats guardrail smoothing
- zero safe capacity means zero actual draw

Review/Rebaseline:
- latest actual debt/collateral/BTC price become revised projection start
- actual income drawn is historical and not double-counted as future income
- strategy change pauses baseline variance
- rebaseline replaces baseline

### Layer 4: golden scenario tests

Golden fixtures must be manually reviewable JSON or TS objects.

Required MVP fixtures:

1. Flat BTC, Fixed Draw, all years safe.
2. Bear path, Fixed Draw, draw becomes constrained.
3. Bear then recovery, Supplemental Guardrail skips/reduces then resumes.
4. Bull path, ARVA increases income over time.
5. Crash path, ARVA Guardrails safety override beats smoothing.
6. Review actuals lower debt because user skipped income; revised projection improves buffer.

Each expected row should include:
- year
- BTC price
- starting debt
- interest
- target draw
- actual draw
- skipped income
- ending debt
- LTV
- drop-to-liquidation
- net BTC after debt
- status

Expected values for small fixtures must be independently calculated in the fixture comments or companion markdown. Do not blindly regenerate expected values from the implementation.

### Layer 5: invariant/property-style tests

Use loops over representative values. Full property-testing library is optional.

Required invariants:
- Higher debt cannot improve LTV if price/collateral unchanged.
- Higher BTC price improves LTV if debt/collateral unchanged.
- Higher required buffer cannot increase available safe draw.
- Higher collateral BTC cannot worsen LTV if debt/price unchanged.
- Actual draw must never exceed available safe draw for safety-capped strategies.
- If all inputs are identical, projection output is identical.
- `skippedIncomeUsd` is never negative.
- No strategy may produce negative debt.

### Layer 6: store/profile tests

Test:
- save/load config
- save/load scenarios
- save/load reviews
- export/import round trip
- invalid import rejected
- schema migration works when introduced
- reset clears local profile only after explicit action

### Layer 7: UI tests

Use React Testing Library for component behavior. Add Playwright later for browser smoke tests.

Required UI tests:
- changing strategy changes strategy-specific inputs
- changing holdings/debt updates dashboard summary
- saving a scenario preserves it after reload of store module
- review form saves snapshot and updates actual history
- import preview shows counts of scenarios/reviews
- strategy change warning appears on Review page

## 3. TDD rule

For engine and store code:

1. Write failing test first.
2. Run it and confirm it fails for the expected reason.
3. Implement minimum code.
4. Run focused test and full relevant suite.
5. Refactor only after green.

No production strategy code without a failing test first.

## 4. Regression test rule

Every discovered bug gets a regression test before the fix.

Example bug class:
- Summary value does not change when strategy/mode changes.

Regression test pattern:
- Build two configs that should produce different outputs.
- Assert downstream summary fields differ.
- Assert selected strategy is reflected in output source.

## 5. Test commands

Expected commands once app exists:

```bash
npm test -- --run
npm run build
```

Engine-focused command:

```bash
npm test -- --run src/engine
```

Store-focused command:

```bash
npm test -- --run src/store
```

UI-focused command:

```bash
npm test -- --run src/components src/pages
```

## 6. Acceptance criteria for test suite

The test suite is acceptable when:

1. It fails if safety caps are removed.
2. It fails if ARVA guardrails override safety.
3. It fails if Supplemental Guardrail borrows during unsafe years.
4. It fails if review actuals are ignored in revised projection.
5. It fails if export/import loses reviews or scenarios.
6. It includes adverse BTC paths, not just flat/growth paths.
7. It includes at least one test where a strategy deliberately draws zero.
8. It includes at least one test where fixed draw and supplemental strategy diverge materially.

## 7. Additional review-driven test requirements

Add these before or during engine implementation.

### 7.1 Negative-control checks

The suite must fail if any of these mutations are made:
- remove safety cap from actual draw
- accrue interest on ending debt instead of starting debt
- let ARVA guardrails override safety cap
- allow Supplemental Guardrail catch-up in MVP
- ignore review actual debt in revised projection

These can be documented manual mutation checks initially; automate later if practical.

### 7.2 Boundary fixtures

Required boundary tests:
- exactly at income LTV ceiling
- exactly at required drop buffer
- exactly at liquidation LTV
- epsilon below/above each threshold
- zero APR
- zero starting debt
- existing debt above max safe debt but below liquidation
- existing debt above liquidation
- ARVA terminal reserve greater than or equal to net equity
- ARVA remaining years = 1
- ARVA assumed return = 0 and near zero

### 7.3 Cross-engine parity

For overlapping semantics, maintain canonical fixtures that can be checked by:
- Python reference model
- generated spreadsheet where practical
- TypeScript web engine

Compare raw row fields, not only summaries.

Do not require parity for ARVA/review behavior until an independent reference exists.

### 7.4 Golden fixture provenance

Each golden fixture should record:
- `calculatedBy`: `manual`, `python-reference`, or `spreadsheet-reference`
- `reviewedAt`
- tolerances per numeric class
- branch expectations: status, reason codes, binding constraint

Include at least one ugly-number fixture with non-round BTC, price, APR, and draw values.

### 7.5 No input mutation

Projection tests must assert the input config is not mutated.
